# Mirage vs Rocjitsu: Architecture Comparison and Intercept Analysis

**Date:** 2026-03-30
**Author:** Stella Laurenzo (with analysis assistance)
**Status:** Working analysis document

---

## Executive Summary

Mirage (powderluv's branch) and rocjitsu (Tony's project) are both AMD GPU
simulators that execute real GPU machine code instruction-by-instruction on the
CPU. They share a common simulation substrate (simdojo) and the same ultimate
goal — simulate AMD GPU hardware in software — but differ significantly in
architecture, maturity, and design philosophy.

**Neither project currently intercepts real HIP/ROCm applications at runtime.**
Both are standalone simulators that accept pre-compiled GPU binaries. The
injection/interception question (containers, KFD shim, hsakmt replacement) is
an integration concern that sits *above* both engines and can be resolved
independently of the simulation core design.

The "GPU acceleration of ISA simulation" question and the "ISA down-translation"
interop question are related but separable: the simulation engine design
determines *how hard* it is to offload inner loops to GPU, but the
down-translation project is a code transformation pass that could feed into
either simulator or run independently.

## 1. What Each Project Actually Is

### Rocjitsu (Tony's project, in rocm-systems)

A **complete, library-grade GPU simulator** with:

- **simdojo**: The original PDES (Parallel Discrete Event Simulation) engine,
  with full component/port/link topology, multi-threaded execution, and both
  functional and cycle-accurate modes
- **Full hardware model**: SoC → IOD → XCD → ShaderEngine → ComputeUnit →
  Wavefront hierarchy, with L1/L2 caches, memory controllers, LDS, command
  processor — all wired as simdojo components communicating via port/link
  message passing
- **ISA engine**: Template-metaprogramming-based instruction hierarchy with
  autogenerated decoders and instruction classes from AMD's Machine-Readable ISA
  XML. Supports CDNA3 (gfx942), CDNA4 (gfx950), and RISC-V (RV32IMAFDC)
- **Public C API**: Reference-counted opaque handles for code loading,
  instruction decoding, VM creation, stepping, and checkpointing
- **Declarative topology**: JSON config with range expansion and link pattern
  mini-language (e.g., `xcd[i].se[j].cu[k]`)
- **Code loading**: Parses HIP fat binaries (Clang offload bundles), extracts
  per-target device ELFs, builds basic blocks via instruction decoding
- **GUI**: Optional visualization via ImGui/imnodes/ImPlot
- **~50K+ lines of C++**, ~312 files, mature test infrastructure

### Mirage (powderluv's branch, in TheRock)

A **simulator-first ROCm development environment** that:

- **Vendors a stripped-down simdojo**: Flat file copy of the core PDES sim/
  files (component, simulation, event_queue, message, topology) without
  rocjitsu's reusable hardware components (cache, register_file, sparse_memory)
- **Reimplements the hardware model**: `SingleGpuSimulator` is a monolithic
  functional execution engine — not a hierarchy of simdojo components. Memory
  is backed by simple host allocations with a linear VA allocator, not a cache
  hierarchy
- **Has its own ISA engine**: Procedural switch-based interpreter with a
  `CompiledOpcode` enum and `CompiledInstruction` struct. Uses a
  decode→compile→cache pipeline where hot kernel launches bypass binary decode
  entirely via write-version-tracked program cache
- **Targets gfx950 (primary), gfx1201 (RDNA4, partial), gfx1250 (stubs)**
- **Python control plane**: Boot/status CLI (`miragectl`, `miraged`), cluster
  profile management, JSON snapshot serialization
- **No C API**: Direct C++ interface via `SingleGpuSimulator`
- **Planned but absent `compat/` layer**: README mentions `compat/` for
  `libhsakmt` but the directory does not exist

## 2. Simdojo: Shared Foundation, Different Usage

Simdojo is the discrete event simulation engine that originated in rocjitsu.
Mirage vendors a copy. The two copies have diverged structurally:

| Aspect | Rocjitsu simdojo | Mirage simdojo |
|--------|-----------------|----------------|
| **Layout** | `include/simdojo/sim/` + `include/simdojo/components/` + `src/` | Flat `third_party/simdojo/` (no subdirs) |
| **Hardware components** | Cache, RegisterFile, SparseMemory, VectorReg, MemoryInterface | Not included |
| **Additional headers** | exec_mode.h, functional.h, pacing_controller.h | clock_domain.h, clocked.h, debug_print.h |
| **Core PDES** | Same design (component, port, link, LBTS barrier) | Same design |

**How rocjitsu uses simdojo:** Every hardware block (CU, cache, CP, memory
controller) is a simdojo `Component` with ports. The full GPU is a topology
graph. Simulation advances via event-driven message passing between components.
This enables cycle-accurate modeling where cache miss latency, memory
controller contention, and dispatch overhead are all visible.

**How mirage uses simdojo:** Only as the *outer event shell* for timed dispatch.
The `SimdojoDispatchSource` queues synthetic dispatch packets at configurable
ticks, the `SimdojoSingleGpuExecutor` receives them and delegates to
`SingleGpuSimulator` for actual execution. The GPU internals (register files,
memory, instruction execution) are **not** simdojo components — they're
procedural C++ inside `SingleGpuSimulator`.

**What this means:** Rocjitsu's hardware model naturally supports cycle-accurate
simulation because cache misses, memory latency, and CU scheduling are
event-driven. Mirage's model is functional-only inside the GPU — simdojo just
provides dispatch timing, not microarchitectural timing.

## 3. ISA Engine Design: Two Very Different Approaches

This is where the projects diverge most significantly.

### Rocjitsu's ISA Engine

**Architecture:** Template metaprogramming + autogenerated class hierarchy.

```
AMD ISA XML → Python codegen → per-instruction C++ classes
                                  ↓
                            class SMovB32Sop1 : public Sop1 {
                              void execute(Wavefront &wf) {
                                sdst.write_scalar(wf, ssrc0.read_scalar(wf));
                              }
                            };
```

- Each instruction is its own class with virtual `execute()` method
- Operands are objects with `read_scalar()`/`write_scalar()`/`read_lane()`/
  `write_lane()` methods that dispatch through the ISA's register file
- Decoder returns `unique_ptr<Instruction>` — heap allocation per decode
- Execution context is the simdojo `Wavefront` component (owns register
  allocation within parent CU's physical register file)
- Supports ISA traits pattern: `IsaTrait<CDNA4>::type` resolves to
  architecture-specific types

**Strengths:** Clean separation of concerns, easy to add individual
instructions, natural for cycle-accurate modeling where each instruction
interacts with the memory hierarchy via the wavefront's CU.

**Weaknesses:** Virtual dispatch overhead per instruction, heap allocation per
decode, deep class hierarchy makes the hot path hard to optimize. The gist
review found ~10K lines of byte-identical code between CDNA3 and CDNA4.

### Mirage's ISA Engine

**Architecture:** Procedural switch-based interpreter with compiled opcode
caching.

```
Binary words → Gfx950BinaryDecoder → DecodedInstruction[]
                                         ↓
                               Gfx950Interpreter::CompileProgram()
                                         ↓
                               CompiledInstruction[] (cached by VA + write_version)
                                         ↓
                               switch (instr.opcode) {
                                 case kSMovB32: sgprs[dst] = sgprs[src]; break;
                                 case kVAddF32: /* vectorized */ break;
                                 ...
                               }
```

- Three-stage pipeline: decode → compile → execute
- `CompiledInstruction` is a flat struct (opcode enum + operand array + flags)
  — no virtual dispatch, no heap allocation in the hot path
- Program cache keyed by `(code_va, word_count)` with write-version
  invalidation — hot kernel re-launches skip decode and compile entirely
- Execution state (`WaveExecutionState`) is a flat struct with inline arrays
  for SGPRs, VGPRs, LDS — not a simdojo component
- Multi-wave barrier sync via scheduler loop (not event-driven)

**Strengths:** Fast functional execution. The compiled-opcode cache and flat
data layout are well-suited for throughput-oriented simulation. The flat
`WaveExecutionState` struct is trivially serializable and could be mapped to
GPU memory.

**Weaknesses:** Monolithic switch statement grows with ISA coverage. No natural
place to insert microarchitectural timing (cache misses, contention). Adding a
new instruction requires touching the decoder, the compiler, and the switch —
three places vs. rocjitsu's one class.

### Coverage Comparison

| Aspect | Rocjitsu | Mirage |
|--------|----------|--------|
| **Primary ISA** | gfx950 (CDNA4) | gfx950 (CDNA4) |
| **Secondary ISAs** | gfx942 (CDNA3), RV32IMAFDC | gfx1201 (RDNA4 partial), gfx1250 (stubs) |
| **Instruction count** | 1242+ catalog, execution for subset | 1242+ catalog, 200+ with execution |
| **DS (LDS) ops** | Partial (review flagged gaps) | Extensive (including D16, pair, wide, atomic RTN) |
| **MFMA** | Stubs (44 silent no-ops flagged) | Not mentioned |
| **Global atomics** | Present but incompletely tested | Full family claimed |
| **Binary decode** | Single pass → Instruction objects | Three-stage: decode → compile → cached execute |

## 4. The Injection Question

**Current state: Neither project intercepts real applications.**

Both require you to pre-compile GPU kernels and feed them to the simulator
explicitly. Neither replaces `/dev/kfd`, provides an `hsakmt` shim, uses
`LD_PRELOAD`, or runs inside a container with device emulation.

Mirage's README references a `compat/` directory for "ROCm-facing compatibility
layers such as `libhsakmt`" but **this directory does not exist yet**. It's
aspirational.

### Why the injection method doesn't matter (much) for the core design

The injection layer sits at the top of the stack:

```
┌─────────────────────────────────────────────────┐
│  HIP Application (unchanged binary)              │
├─────────────────────────────────────────────────┤
│  Injection Layer (one of the options below)       │
│  ┌─────────────────────────────────────────────┐ │
│  │ Option A: LD_PRELOAD libhsakmt shim         │ │
│  │ Option B: Container with /dev/kfd emulation │ │
│  │ Option C: Modified ROCr/HSA runtime         │ │
│  │ Option D: FUSE filesystem for /dev/kfd      │ │
│  └─────────────────────────────────────────────┘ │
├─────────────────────────────────────────────────┤
│  Translation Layer                                │
│  (dispatch packets → simulator dispatch packets)  │
├─────────────────────────────────────────────────┤
│  Simulation Engine (rocjitsu or mirage core)      │
└─────────────────────────────────────────────────┘
```

The injection method determines *how* you capture the application's GPU
dispatches. The simulation engine determines *what you do with them*. These are
separable concerns. You could:

- Use rocjitsu's cycle-accurate hardware model with any injection method
- Use mirage's fast functional interpreter with any injection method
- Swap injection methods without touching the simulation engine

### What actually matters for injection

The injection layer needs to handle:

1. **Device enumeration** — Report simulated GPU topology to the application
   (rocm_agent_enumerator, sysfs, topology queries)
2. **Memory management** — Intercept `hsakmt` allocation calls, map them to
   simulator memory (this is where "problems with mmap" come from — the shim
   needs to provide GPU-visible virtual addresses that the simulator can
   resolve)
3. **Queue creation and dispatch** — Intercept AQL packet submission, translate
   to simulator dispatch packets
4. **Signal/fence management** — Intercept completion signals so the
   application's synchronization works
5. **Code object loading** — Intercept `hsa_code_object_reader` to extract
   kernels for the simulator's ISA engine

The simplest viable approach is probably an `hsakmt` replacement library
(`LD_PRELOAD` or link-time substitution) because `hsakmt` is the narrowest
waist in the ROCm stack — it's the boundary between user-space and the kernel
driver. Everything above it (ROCr, HIP runtime) doesn't need to change.

Container-based approaches add operational complexity without fundamentally
changing what the shim needs to do — you still need to implement the same
`hsakmt` API surface, you're just choosing where the library boundary lives.

### Bottom line on the injection debate

**This is a detail that comes out in the wash.** The hard engineering is in the
simulation engine (ISA correctness, memory model, performance). The injection
layer is a well-understood problem — it's essentially what every GPU emulator
and virtualization project does. The choice between LD_PRELOAD, container, and
modified runtime is an operational/deployment decision, not an architectural
one. All three approaches require implementing the same ~50 hsakmt API entry
points.

## 5. GPU Acceleration of ISA Simulation

This is the question: can the ISA simulation engine itself run on a GPU to go
faster?

### What "GPU-accelerated ISA simulation" means

The inner loop of both simulators is: decode instruction → read registers →
compute → write registers → advance PC. This is inherently serial *per
wavefront* (each instruction depends on the previous one's register state).

However, a real GPU workload has **many wavefronts executing concurrently**.
If you're simulating a kernel with 1024 wavefronts, each wavefront's
instruction stream is independent (modulo barriers and atomics). This is
embarrassingly parallel — you could simulate each wavefront on a separate
GPU thread.

### Mirage's design is closer to GPU-acceleratable

Mirage's `WaveExecutionState` is a flat struct:
```cpp
struct WaveExecutionState {
  array<uint32_t, 128> sgprs;           // 512 bytes
  array<array<uint32_t, 64>, 128> vgprs; // 32KB
  array<byte, 65536> lds_bytes;          // 64KB
  uint64_t exec_mask, vcc_mask, pc;
  bool halted, waiting_on_barrier, scc;
};
```

The `CompiledInstruction` is a flat struct:
```cpp
struct CompiledInstruction {
  CompiledOpcode opcode;        // uint16_t enum
  array<InstructionOperand, 8> operands;
  uint8_t operand_count, flags, register_dword_count, ...;
};
```

This data layout could be mapped to GPU memory. The compiled program is a flat
array of `CompiledInstruction`. The execution loop is a switch statement over
an enum. This is structurally similar to a GPU-side interpreter — you could
write a HIP kernel that:

1. Loads a `WaveExecutionState` per GPU thread
2. Loads the shared `CompiledInstruction[]` program
3. Runs the switch-based interpreter loop
4. Writes back the modified `WaveExecutionState`

The main challenges:
- **Memory access**: Simulated memory loads/stores need to go through the
  simulator's memory model. On GPU, this means the simulated address space
  must be accessible (e.g., mapped to a large GPU buffer)
- **Barriers**: S_BARRIER requires cross-wavefront synchronization within a
  workgroup. On GPU, this maps naturally to `__syncthreads()` if simulated
  wavefronts within a workgroup are mapped to threads within a GPU workgroup
- **Divergent control flow**: The switch statement is divergent across simulated
  wavefronts executing different instructions. GPU execution will serialize
  across different opcodes. This is acceptable if most wavefronts are at
  similar points in the program (common for SIMT workloads)

### Rocjitsu's design is harder to GPU-accelerate

Rocjitsu's execution model involves:
- Virtual dispatch (`Instruction::execute()`) — no GPU equivalent
- Heap-allocated instruction objects with pointer-chasing
- Wavefronts that are simdojo components with port-based communication
- Event queues, LBTS barriers, cross-partition message passing

This is fundamentally a pointer-heavy, object-oriented, event-driven
architecture. It's excellent for cycle-accurate modeling but structurally
incompatible with GPU execution.

### Practical path to GPU-accelerated simulation

**Phase 1 (functional, GPU-accelerated):** Take mirage's compiled-opcode
interpreter, port the switch statement to a HIP kernel. Each GPU thread
simulates one wavefront. Memory is a large GPU-side buffer. No timing model.
This gives you fast functional simulation for correctness testing.

**Phase 2 (timing, CPU-side):** Use rocjitsu's simdojo-based hardware model
for cycle-accurate analysis on CPU. Or extend mirage's simdojo integration to
model more microarchitectural detail.

**Phase 3 (timing, GPU-assisted):** The functional GPU kernel produces
execution traces (instruction counts, memory access patterns). Feed these
traces into the timing model on CPU. This is the standard
"functional-first, timing-directed" simulation methodology used by academic
architecture simulators (gem5, SST, etc.).

## 6. ISA Down-Translation Interop

The question: how does an ISA simulation engine interop with a project that
does online down-translation from newer→older GPU ISA (emulating gfx1250 on
gfx1201 hardware, for example)?

### What down-translation produces

An ISA down-translator takes a kernel compiled for ISA version N+1 and
produces an equivalent kernel for ISA version N. The output is a
**rewritten code object** — same semantics, different instruction encoding.

### Integration points

There are two ways the simulator and down-translator interact:

#### Option A: Down-translate, then run on real hardware

```
gfx1250 kernel → down-translator → gfx1201 kernel → real gfx1201 GPU
```

The simulator's role here is **validation**: you run the original gfx1250
kernel in the simulator and the down-translated gfx1201 kernel on real
hardware, then compare results. The simulator provides the reference
implementation.

#### Option B: Down-translate, then simulate

```
gfx1250 kernel → down-translator → gfx1201 kernel → gfx1201 simulator
```

This is useful when you don't have real gfx1201 hardware either. The
simulator validates the down-translator's output.

#### Option C: Simulate the source ISA directly

```
gfx1250 kernel → gfx1250 simulator (mirage has stubs for this)
```

If the simulator supports the source ISA, you don't need down-translation
at all for validation purposes. You only need down-translation for running
on real hardware.

### Where down-translation fits architecturally

The down-translator is a **code transformation pass** that operates on code
objects before they reach the simulation engine. It's analogous to a compiler
backend pass:

```
Code Object (ISA N+1)
    ↓
Down-Translator (ISA N+1 → ISA N)
    ↓
Code Object (ISA N)
    ↓
Simulation Engine (ISA N) OR Real Hardware (ISA N)
```

Both mirage and rocjitsu accept code objects as input. The down-translator
doesn't need to know about the simulator's internals — it just needs to
produce valid code objects for the target ISA.

### GPU acceleration of down-translation

The down-translator itself could potentially be GPU-accelerated if it
operates on many kernels or basic blocks in parallel. But single-kernel
translation is likely fast enough on CPU (it's essentially a compiler pass
on a few KB of code). The bottleneck is correctness, not throughput.

The more interesting GPU acceleration angle is: if you're doing
**online** (runtime) down-translation, you want the translation to be fast
enough to not block kernel launch. If the simulator is already running on GPU
(Phase 1 above), you could do translation on CPU while the previous kernel
is still simulating on GPU — pipeline the two.

## 7. Stakeholder Map and Actual Requirements

The different projects reflect different stakeholders solving different
problems. The risk is that without a common backplane, we end up with half a
dozen disconnected, divergent projects that each re-derive the same ISA
knowledge, the same code loading, and the same memory model.

### Tony / Rocjitsu: System-level simulation

**Primary concern:** Cycle-accurate modeling of the *system*, not just the
shader core. CP firmware, packet processing, RISC-V microcontrollers on the
GPU, memory controller behavior, cache coherence protocol. These are exactly
the subsystems that functional-only simulation tends to abstract away, but
they are critical for architecture group workflows.

**Why rocjitsu looks the way it does:** The simdojo component model exists
because *every block on the chip* needs to be a first-class simulation entity
with its own timing. CUs, caches, command processors, memory controllers, IODs
— they all communicate via timed message passing because that's how the real
hardware works. The RISC-V simulator isn't a curiosity — most processors on
AMD GPUs are RISC-V cores. If you're modeling the CP firmware path, you need
to simulate the RISC-V core that runs it.

**What he needs from a backplane:** Shared ISA knowledge (decoder tables,
instruction metadata, operand semantics) and shared code loading (ELF parsing,
offload bundle extraction) so his team isn't maintaining a parallel universe of
AMD ISA infrastructure. The simulation engine itself (simdojo + hardware model)
is purpose-built for his use case and shouldn't be forced into a different
shape.

### Anush / Mirage: Fastest path to "run future code now"

**Primary concern:** Emulate the next GPU generation on the current one, as
early as possible, with correctness. Speed of functional simulation matters.
There is interest in eventually adding timing accuracy, though it's worth
noting that functional-first simulators face a well-known gap between
functional correctness and the cycle-level fidelity required for performance
tuning of optimized kernels. Closing that gap would require significant
investment in microarchitectural modeling.

**Why mirage looks the way it does:** The compiled-opcode interpreter with flat
data structures is optimized for throughput. The program cache exists because
the use case is "run the same kernel thousands of times" not "model what
happens inside the cache hierarchy on each run." Multi-arch stubs (gfx1201,
gfx1250) exist because the whole point is to simulate hardware that doesn't
exist yet.

**What he needs from a backplane:** Shared ISA decoders and instruction
metadata so that adding gfx_N+1 support doesn't require building a parallel
codegen pipeline. Shared code object loading. A clean interface to the
down-translator so he can transpile→execute without caring which simulator
backend runs the result.

### The down-translation project: Code transformation, not simulation

This is a compiler pass that rewrites code objects from ISA N+1 → ISA N.
It doesn't simulate anything. It doesn't model hardware. It transforms
instruction streams. Its relationship to both simulators is: it produces
code objects that either simulator can consume, or that real hardware can
execute.

### The integration problem: One backplane, multiple backends

The actual problem isn't "which simulator wins." It's that without shared
infrastructure, every project re-derives:

- ISA decoder tables from the same AMD XML specs
- Code object loading (ELF parsing, offload bundle extraction)
- Instruction metadata and operand semantics
- Memory model abstractions
- The injection/interception layer (when someone finally builds it)

Each re-derivation diverges immediately and creates maintenance burden that
scales with the number of projects × number of ISA generations.

## 8. The Backplane: What It Would Look Like

The backplane isn't a simulator. It's the shared infrastructure that all
simulation, emulation, and analysis tools build on.

```
┌─────────────────────────────────────────────────────────┐
│                    User-Facing Tools                     │
│                                                         │
│  Mirage           Rocjitsu          Down-Translator     │
│  (fast func sim)  (cycle-accurate)  (ISA N+1 → N)      │
│                                                         │
├─────────────────────────────────────────────────────────┤
│                    Backplane                             │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────┐ │
│  │ ISA Registry │  │ Code Loading │  │ Injection /   │ │
│  │              │  │              │  │ Interception  │ │
│  │ • Decoders   │  │ • ELF parse  │  │               │ │
│  │ • Metadata   │  │ • Offload    │  │ • hsakmt shim │ │
│  │ • Operand    │  │   bundles    │  │ • Device enum │ │
│  │   semantics  │  │ • Code obj   │  │ • Memory mgmt │ │
│  │ • Codegen    │  │   registry   │  │ • Queue/signal│ │
│  │   from XML   │  │              │  │               │ │
│  └──────────────┘  └──────────────┘  └───────────────┘ │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐                     │
│  │ simdojo      │  │ Memory Model │                     │
│  │ (PDES core)  │  │ Abstraction  │                     │
│  │              │  │              │                     │
│  │ • Component  │  │ • VA space   │                     │
│  │ • Port/Link  │  │ • Regions    │                     │
│  │ • Simulation │  │ • Load/Store │                     │
│  │   engine     │  │   interface  │                     │
│  └──────────────┘  └──────────────┘                     │
├─────────────────────────────────────────────────────────┤
│                    AMD ISA XML Specs                     │
│                    (single source of truth)              │
└─────────────────────────────────────────────────────────┘
```

### Backplane components

**ISA Registry** — One codegen pipeline from AMD's Machine-Readable ISA XML.
Produces decoder tables, instruction metadata, operand semantics, and encoding
specs that all consumers use. Currently both projects have their own:
rocjitsu has `lib/python/amdisa/codegen.py` (229KB), mirage has
`tools/generate_gfx950_isa_catalog.py`. These should converge.

**Code Loading** — One ELF parser, one offload bundle extractor, one code
object abstraction. Rocjitsu's is more mature (full fat binary support).
Mirage's is minimal (raw binary words). The backplane version should handle
the full stack.

**simdojo** — One copy, properly packaged as a library. Both projects depend
on it; neither should vendor a fork. Rocjitsu uses it deeply (every hardware
block is a component). Mirage uses it lightly (dispatch timing only). Both
uses are valid.

**Memory Model Abstraction** — A common interface for load/store that both
simulators implement. Mirage's `ExecutionMemory` and rocjitsu's memory
pipeline serve different fidelity levels but could share the interface.

**Injection / Interception** — Built once, delegates to whichever backend.
Targets the `hsakmt` API surface. Doesn't exist in either project today.

### What this enables

- **Add ISA gfx_N+1:** Update the XML spec, run one codegen pipeline, all
  tools get decoder support simultaneously.
- **Tony adds a new hardware block:** It's a simdojo component. It uses the
  shared ISA decoder. Mirage doesn't need to know about it.
- **Anush adds fast-functional for a new ISA:** He uses the shared decoder
  tables to build his compiled-opcode mapping. Rocjitsu doesn't need to change.
- **Down-translator evolves:** It consumes and produces code objects via the
  shared code loading layer. Both simulators can validate its output.
- **Someone finally builds the hsakmt shim:** It delegates to mirage for fast
  emulation or rocjitsu for cycle-accurate analysis, selected at runtime.

### What this does NOT do

- Force either simulator to change its execution architecture
- Make rocjitsu's cycle-accurate model fast-functional
- Make mirage's functional model cycle-accurate
- Merge the two projects into one

The simulators stay separate because they serve different purposes. The
backplane eliminates the *duplicated infrastructure* that makes them diverge
on things they shouldn't be diverging on.

## 9. Concrete Next Steps

### Near-term (shared ISA infrastructure)

1. **Factor simdojo into a standalone CMake package** with `find_package()`
   support. Both projects depend on it. Stop vendoring.

2. **Converge the ISA codegen pipelines.** Rocjitsu's `codegen.py` is more
   mature. Mirage's `generate_gfx950_isa_catalog.py` produces a subset. The
   output formats differ but the input (AMD XML) is the same. Define a common
   intermediate representation (instruction metadata, decoder tables) that both
   projects consume, even if they generate different final C++ from it.

3. **Share the code object loading layer.** Rocjitsu's ELF/offload-bundle
   parser should be the one implementation. Mirage's synthetic dispatch can
   sit on top of it.

### Medium-term (backplane interfaces)

4. **Define the memory model interface.** Both projects have one
   (`ExecutionMemory` / memory pipeline). Agree on the abstract interface so
   ISA execution code can be written once against the interface and plugged
   into either simulator's memory backend.

5. **Define the dispatch interface.** A common dispatch packet format that
   both simulators accept. Mirage's `SyntheticDispatchPacket` and rocjitsu's
   `DispatchPacket` serve the same purpose. Unify.

6. **Build the hsakmt shim** as a separate library that translates ROCm
   runtime calls into dispatch packets against the common interface.

### Longer-term (composability)

7. **ISA execution as a pluggable backend.** The backplane defines "given
   this code object and this wave state, execute and return results." Mirage
   provides the fast-functional backend. Rocjitsu provides the cycle-accurate
   backend. The down-translator provides the "transpile and run on real
   hardware" backend. Same interface, different implementations.

8. **Mirage's compiled-opcode path as a GPU-accelerated backend.** If/when
   someone ports the switch-based interpreter to HIP, it slots in as another
   backend behind the same interface.

### Rack-scale (builds on backplane)

9. **Distributed transport for simdojo.** Replace shared-memory
   `CrossPartitionQueue` with a network-backed implementation. Add
   distributed LBTS reduction. This is the enabler for multi-process,
   multi-machine simulation.

10. **XGMI and RDMA fabric models** as simdojo components. Fast-functional
    (fixed-delay pipe) and cycle-accurate (flow control, congestion) variants.

11. **Cluster topology configuration.** Extend rocjitsu's JSON topology DSL
    to describe multi-node, multi-GPU configurations with fabric links.

12. **Host trace replay.** Capture real application host-side API calls and
    replay them into the distributed simulator as dispatch packets, avoiding
    the need for a full CPU model.

## 10. The Transpiler Question, Revisited

"If we had a cross-gfx-arch transpiler, could we plug it into the system
simulator and replace the wave-level ISA interpreter with real GPU execution?"

**Yes — with a shared backplane, this is exactly the right decomposition.**

The analogy is the Android emulator's GPU mode. Android's emulator runs the
full system (ARM CPU, kernel, drivers, firmware) in QEMU. For GPU work, it
offers two modes:

1. **Full simulation** — GPU commands are interpreted instruction-by-instruction
   inside the emulator. Accurate but slow.
2. **Host GPU passthrough** — GPU commands are translated and dispatched to the
   host machine's real GPU. Fast, but you're not simulating the GPU
   microarchitecture.

The same decomposition applies here:

```
┌─────────────────────────────────────────────────┐
│  System Simulator (always running)               │
│                                                  │
│  Command Processor (RISC-V firmware emulation)   │
│  Packet processing / dispatch logic              │
│  Memory controllers, cache hierarchy, IODs       │
│  Queue management, signal/fence handling         │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │  Wave Execution (swappable backend)        │  │
│  │                                            │  │
│  │  Option A: ISA interpreter (rocjitsu CU)   │  │
│  │  Option B: ISA interpreter (mirage fast)   │  │
│  │  Option C: Transpile + dispatch to real GPU│  │
│  └────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

The system simulator continues to model everything *around* the shader cores:
CP firmware, packet processing, memory hierarchy, dispatch timing. The wave
execution backend is the part that gets swapped. In option C, the transpiler
converts the kernel from ISA N+1 → ISA N, and real hardware executes it. The
system simulator still models the dispatch path, memory traffic patterns, and
completion signaling — you just skip the instruction-by-instruction
interpretation of the shader program itself.

### Which project is architecturally closer to this?

**Rocjitsu.** Its simdojo component model already separates the system-level
simulation (CP, caches, memory controllers, IODs) from the wave execution
(ComputeUnit → Wavefront → ISA interpreter). The ComputeUnit is a simdojo
component with ports. You could replace what happens *inside* the CU's
`execute()` path — currently "decode and interpret instructions via the ISA
engine" — with "transpile and dispatch to real GPU, wait for completion,
update memory state." The rest of the system simulation (event-driven,
timed message passing between all other components) continues unchanged.

Mirage's `SingleGpuSimulator` is monolithic — the ISA interpreter, memory
management, wave scheduling, and dispatch logic are all interleaved in one
class. Swapping out just the wave execution would require significant
refactoring to separate it from the surrounding logic.

This doesn't mean mirage can't get there, but rocjitsu's existing component
boundaries already provide the seams where a transpiler backend would plug in.

### What the transpiler does NOT replace

Even in option C, the system simulator is still doing significant work:

- **CP firmware emulation** — The command processor parses AQL packets,
  manages queues, dispatches to CUs. This runs on a RISC-V core in real
  hardware. The simulator models this.
- **Memory hierarchy** — Cache coherence, memory controller scheduling, HBM
  bandwidth modeling. Even if shader execution happens on real hardware, the
  system simulator can model the memory traffic patterns.
- **Dispatch timing** — How long it takes from doorbell ring to first wave
  launch. This is where real performance problems often hide.
- **Multi-GPU fabric** — IOD-to-IOD links, XGMI topology, cross-GPU memory
  traffic.

The wave ISA interpreter is the most expensive part of simulation (it runs
the most instructions), which is exactly why it's the part worth offloading.
But it's not the *only* part, and the system-level modeling that remains is
precisely what the architecture group needs.

## 11. Rack-Scale Simulation

### The requirement

A virtual rack-scale simulator modeling a full GPU cluster: multiple nodes,
each with multiple GPUs, connected via XGMI (intra-node), PCIe (CPU-GPU),
and RDMA/RoCE (inter-node). The same fidelity spectrum applies at every
level — you want both fast-functional and cycle-accurate modes for the
interconnect fabric, not just for the shader cores.

### Why this is a backplane problem, not a new project

A rack-scale simulator is not a different kind of thing from a single-GPU
simulator. It's the same simulation, with more components and longer-latency
links between them. The question is whether the architecture supports
composition at that scale.

**Monolithic architectures (mirage's `SingleGpuSimulator`) do not scale to
this.** A monolithic simulator encapsulates one GPU's state in one object. To
simulate 8 GPUs on a node, you'd need 8 instances. To model the XGMI fabric
between them, you'd need to bolt on an ad-hoc communication layer. To
distribute across physical machines, you'd need to invent a remoting
protocol. Each of these is a significant engineering effort because the
original design didn't anticipate composition.

**Component-based architectures (rocjitsu's simdojo model) scale naturally.**
Each GPU is already a hierarchy of simdojo components. A multi-GPU node is a
CompositeComponent containing N GPU composites plus XGMI link components.
A cluster is a CompositeComponent containing M node composites plus network
fabric components. The simulation engine doesn't change — it just processes
a larger topology graph.

### The russian doll

The fidelity spectrum applies at every level of the hierarchy:

```
┌─ Cluster ──────────────────────────────────────────────────────┐
│                                                                │
│  ┌─ Node 0 ──────────────────┐  ┌─ Node 1 ──────────────────┐ │
│  │                            │  │                            │ │
│  │  ┌─ GPU 0 ─┐ ┌─ GPU 1 ─┐ │  │  ┌─ GPU 0 ─┐ ┌─ GPU 1 ─┐ │ │
│  │  │ CP      │ │ CP      │ │  │  │ CP      │ │ CP      │ │ │
│  │  │ CU[]    │ │ CU[]    │ │  │  │ CU[]    │ │ CU[]    │ │ │
│  │  │ L2/HBM  │ │ L2/HBM  │ │  │  │ L2/HBM  │ │ L2/HBM  │ │ │
│  │  └────┬────┘ └────┬────┘ │  │  └────┬────┘ └────┬────┘ │ │
│  │       │   XGMI    │      │  │       │   XGMI    │      │ │
│  │       └─────┬─────┘      │  │       └─────┬─────┘      │ │
│  │             │ PCIe        │  │             │ PCIe        │ │
│  │         [CPU/Host]        │  │         [CPU/Host]        │ │
│  └─────────────┬─────────────┘  └─────────────┬─────────────┘ │
│                │         RDMA/RoCE             │               │
│                └───────────────┬───────────────┘               │
│                            [Network]                           │
└────────────────────────────────────────────────────────────────┘
```

At each level, the same choice exists:

| Level | Fast-functional | Cycle-accurate |
|-------|----------------|----------------|
| **Wave execution** | Compiled-opcode interpreter | Full CU pipeline model |
| **Intra-GPU memory** | Flat memory (instant) | L1/L2 cache hierarchy with coherence |
| **XGMI (intra-node)** | Latency annotation only | Link-level flow control, credit model |
| **RDMA (inter-node)** | Message passing with fixed delay | NIC model, congestion, routing |
| **CP / dispatch** | Instant dispatch | RISC-V firmware emulation |

A well-designed backplane lets you mix fidelity levels independently at each
layer. You might want cycle-accurate XGMI modeling with fast-functional wave
execution (to study fabric contention without paying the cost of interpreting
every instruction), or fast-functional everything (to validate multi-node
correctness quickly).

### Why simdojo is the right substrate

Simdojo's PDES architecture was designed for exactly this kind of
hierarchical, distributed simulation. Key properties that map to rack-scale:

**Partitioning.** Simdojo includes a multilevel Fiduccia-Mattheyses graph
partitioner that assigns components to threads by minimizing cross-partition
communication while balancing load. For rack-scale simulation, this same
partitioner assigns components to *processes* across a cluster. Components
within a partition communicate via local event queues; components across
partitions communicate via cross-partition message passing — which in the
distributed case becomes network I/O rather than shared-memory queues.

**LBTS synchronization.** The barrier-based Lower Bound on Timestamp protocol
already handles the fundamental problem of distributed simulation: ensuring
no partition processes events from the future. The multi-threaded
implementation uses `std::barrier` with a completion function that computes
global LBTS. The distributed version replaces this with an MPI allreduce or
similar collective — same algorithm, different transport.

**Async event injection.** External threads can inject events via
`schedule_event_async()` with thread-safe buffering and timestamp flooring.
In a distributed simulator, this is how remote events arrive: the network
receive thread injects them into the local partition's async queue.

**Wall-clock pacing.** Simdojo includes a PI-controlled pacing servo with
token-bucket burst absorption, designed for real-time co-simulation. In a
distributed setting, this keeps partitions synchronized with wall time,
preventing fast partitions from racing ahead and consuming memory with
buffered cross-partition messages.

**Component hierarchy.** The CompositeComponent model means a "node" is just
a composite containing GPU composites, CPU composites, and fabric link
components. A "cluster" is a composite containing node composites and network
components. The simulation engine doesn't care about the nesting depth.

### What needs to be built

The pieces that don't exist yet, in order of dependency:

**1. Distributed transport for cross-partition events.**
Simdojo's `CrossPartitionQueue` is currently shared-memory. A network-backed
implementation (MPI, gRPC, RDMA verbs, or even TCP) would allow partitions
to run on different physical machines. The LBTS barrier needs a distributed
reduction (MPI_Allreduce is the standard choice). This is well-understood
distributed simulation infrastructure — not novel research.

**2. Fabric models (XGMI, PCIe, RDMA/RoCE).**
These are simdojo components with ports. An XGMI link model accepts memory
requests on one port and delivers them on another with appropriate latency
and bandwidth constraints. The fast-functional version is a fixed-delay pipe.
The cycle-accurate version models credit-based flow control, link-level
protocols, and contention. Both are standard network simulation components.

**3. Multi-GPU memory coherence.**
Within a node, GPUs share a coherence domain via XGMI. The coherence protocol
(currently modeled within a single XCD in rocjitsu) needs to extend across
GPUs. This is the most architecturally complex piece because AMD's multi-GPU
coherence protocol has subtleties around memory types, atomic scope, and
fine-grained vs. coarse-grained coherence.

**4. Host CPU model (optional).**
For full-system simulation, you need a model of the host CPU issuing
dispatches, managing memory, and running the HIP runtime. This can range from
"replay a recorded trace of host-side API calls" (functional) to a full CPU
simulator (overkill for most use cases). The trace-replay approach is
standard practice in GPU architecture simulation.

**5. Cluster topology configuration.**
Extend rocjitsu's declarative JSON topology config to describe multi-node
clusters. The existing range-expansion and link-pattern mini-language
(`xcd[i].se[j].cu[k]`) naturally extends to `node[n].gpu[g].xcd[i]...`.

### How the backplane enables this

Without a backplane, building a rack-scale simulator means choosing one of:

- **Fork rocjitsu and add multi-GPU** — gets you the component model but
  you're maintaining a fork
- **Build from scratch** — re-derive everything (ISA, memory model, dispatch)
  for the Nth time
- **Bolt onto mirage** — requires fundamental restructuring of the monolithic
  architecture

With a shared backplane:

- **simdojo** provides the simulation substrate (already multi-threaded,
  already has partitioning, needs distributed transport)
- **ISA registry** provides decoder tables for all GPU targets (shared across
  all fidelity levels)
- **Code loading** handles fat binary extraction (shared across all tools)
- **Wave execution backends** plug in per fidelity requirement (rocjitsu CU
  model, mirage fast interpreter, or transpiler+real GPU)
- **Fabric models** are new simdojo components that snap into the topology
- **The injection layer** captures real application traffic and replays it
  into the distributed simulator

Each piece is independent. Teams can work on fabric models without touching
ISA execution. The RDMA model can be developed and tested in isolation, then
wired into the cluster topology.

### Scaling expectations

Some rough numbers to set expectations on what "simulate a GPU cluster" means:

| What you're simulating | State per unit | Scale |
|----------------------|---------------|-------|
| Wavefront (registers) | ~33KB (128 SGPRs + 128×64 VGPRs + 64KB LDS) | 1000s per GPU |
| Compute Unit | ~512KB (register files + caches) | 128-256 per GPU |
| GPU (all CUs + caches + memory) | ~100MB-1GB (depends on HBM model) | 8 per node |
| Node (8 GPUs + host) | ~1-8GB | 10s-1000s per cluster |
| Cluster (N nodes) | N × node size | limited by physical hosts |

Fast-functional simulation (no cache state, flat memory) is dramatically
cheaper — perhaps 10-100x less state per GPU. This is where the fidelity
dial matters most for scalability.

For cycle-accurate simulation, the standard approach is to distribute:
assign 1-2 simulated GPUs per physical CPU core, use the partitioner to
minimize cross-boundary traffic, and accept that simulation will run
10,000-100,000x slower than real hardware. This is consistent with every
other cycle-accurate GPU simulator (gem5-gpu, GPGPU-Sim, SST + Balar).

For fast-functional simulation, you can simulate many more GPUs per physical
core because you're not modeling the microarchitecture. The bottleneck shifts
to memory bandwidth (for the simulated address space) and interconnect
modeling.

## 12. Concrete Proposal: Simulation Backplane with Pluggable Wave Backends

### The architectural seam

Rocjitsu's `ComputeUnit` is a simdojo component. Its `step()` method
currently does: pick a ready wavefront slot, decode an instruction via the
ISA class hierarchy, execute it, interact with the L1/L2 caches via timed
port messages, advance the wavefront state, check for completion.

That method is the seam. Everything *outside* it — the command processor
dispatching work to CUs, the cache hierarchy servicing memory requests, the
memory controllers managing HBM bandwidth, the XGMI links routing cross-GPU
traffic — is system-level simulation that runs identically regardless of how
the wave execution is implemented.

The proposal is to formalize this seam as a **wave execution backend
interface** within the rocjitsu/simdojo simulation stack, with three initial
backends.

### The interface

```cpp
/// Abstract interface for wave execution within a ComputeUnit.
///
/// The CU component owns wavefront slots and register allocations.
/// The backend is responsible for executing instructions and reporting
/// memory side effects. The CU handles scheduling, dispatch, and
/// interaction with the surrounding memory hierarchy.
class WaveExecutionBackend {
public:
  virtual ~WaveExecutionBackend() = default;

  /// Execute up to `max_instructions` on the given wavefront.
  /// Returns the execution result (completed, yielded, blocked on
  /// barrier, blocked on waitcnt, etc.).
  ///
  /// Memory side effects are reported via the MemoryTraceCallback
  /// if one is registered. The CU uses these to drive cache/memory
  /// simulation even when the backend itself uses flat memory.
  virtual WaveExecResult execute(
      WavefrontState& wf,
      const CodeObject& code,
      uint32_t max_instructions,
      MemoryTraceCallback* trace) = 0;

  /// Prepare a code object for execution. Backends may use this
  /// for ahead-of-time compilation, program caching, or transpilation.
  virtual void prepare(const CodeObject& code) {}

  /// Query backend capabilities.
  virtual BackendCaps capabilities() const = 0;
};

/// Memory access record emitted by the backend during execution.
/// The surrounding CU/cache components consume these to model
/// memory hierarchy behavior.
struct MemoryTraceEntry {
  uint64_t address;
  uint32_t size_bytes;
  MemoryOpKind kind;       // LOAD, STORE, ATOMIC, LDS
  MemoryScope scope;       // WAVEFRONT, WORKGROUP, DEVICE, SYSTEM
  MemoryOrder order;       // RELAXED, ACQUIRE, RELEASE, ACQ_REL, SEQ_CST
};

/// Callback interface for memory trace collection.
class MemoryTraceCallback {
public:
  virtual void on_memory_op(const MemoryTraceEntry& entry) = 0;
};

/// What the backend can and cannot do.
struct BackendCaps {
  bool provides_timing;        // Does execute() model pipeline timing?
  bool provides_memory_trace;  // Does execute() emit MemoryTraceEntries?
  bool modifies_real_memory;   // Does execute() have real HW side effects?
  bool supports_checkpoint;    // Can wavefront state be saved/restored?
};
```

### Backend 1: Rocjitsu cycle-accurate (existing)

This is what rocjitsu does today, refactored behind the interface.

```
CU.step() → WaveExecutionBackend::execute()
              │
              ├─ Decode instruction (per-class virtual dispatch)
              ├─ Execute (operand read/write through register file)
              ├─ Emit MemoryTraceEntry for loads/stores
              │    → CU routes to L1 cache component via timed port
              │    → L1 miss → L2 via timed port
              │    → L2 miss → memory controller via timed port
              ├─ Wait counter management (vmcnt, lgkmcnt, expcnt)
              └─ Return WaveExecResult (yielded after 1 instruction)
```

**BackendCaps:** `{timing: true, memory_trace: true, real_memory: false,
checkpoint: true}`

This backend executes one instruction per `step()` call, allowing the
simdojo event loop to interleave cache responses, CP events, and cross-GPU
traffic between instructions. This is what makes it cycle-accurate.

### Backend 2: Mirage fast-functional

Mirage's compiled-opcode interpreter, adapted to the backend interface.

```
CU.step() → WaveExecutionBackend::execute(max_instructions=ALL)
              │
              ├─ Look up CompiledInstruction[] in program cache
              │    (cache hit: skip decode+compile entirely)
              ├─ Run switch-based interpreter loop
              │    ├─ Flat register access (array indexing, no port messages)
              │    ├─ Flat memory access (host memory, no cache model)
              │    ├─ If trace callback registered:
              │    │    emit MemoryTraceEntry per load/store
              │    ├─ Barrier → yield with kBlockedOnBarrier
              │    └─ S_ENDPGM → yield with kCompleted
              └─ Return WaveExecResult
```

**BackendCaps:** `{timing: false, memory_trace: optional, real_memory: false,
checkpoint: true}`

This backend runs the entire wavefront to completion (or barrier) in a single
`execute()` call. The CU component sees a completed wavefront and processes
the aggregated memory trace (if requested) through the cache hierarchy. This
gives you system-level timing (dispatch overhead, fabric contention) without
per-instruction microarchitectural detail.

**The memory trace question.** This is the key design decision. There are
three modes:

1. **No trace** (`trace = nullptr`). Fastest. The surrounding memory hierarchy
   sees no traffic from wave execution. Useful for pure correctness testing
   or when you only care about dispatch/fabric timing, not memory bandwidth.

2. **Post-hoc trace.** The fast-functional backend records every load/store
   address and size during execution. After the wave completes, the CU
   replays the trace through the cache hierarchy. This gives approximate
   memory traffic modeling — the addresses are correct, but the timing is
   compressed (all accesses appear to happen at once, not spread across the
   execution timeline).

3. **Periodic yield.** The backend executes N instructions, yields with a
   partial trace, the CU processes the trace through caches, then resumes
   the backend for the next N instructions. This gives better timing
   distribution at the cost of more yield/resume overhead. N is a tuning
   knob: N=1 approaches cycle-accurate behavior; N=∞ is pure fast-functional.

Option 2 is the practical starting point. Option 3 is the refinement if
memory traffic timing matters.

### Backend 3: Hotswap (transpiler + real GPU)

The transpiler converts the kernel from ISA N+1 to ISA N, dispatches to real
GPU hardware, and returns the results.

```
CU.step() → WaveExecutionBackend::execute()
              │
              ├─ prepare() already transpiled ISA N+1 → ISA N
              ├─ Marshal wavefront state to GPU-accessible memory
              │    (SGPRs, VGPRs, kernel args, global memory)
              ├─ Submit transpiled kernel to real GPU queue
              ├─ Wait for completion
              ├─ Unmarshal wavefront state back
              ├─ Memory trace: not available (real HW side effects)
              │    → statistical model or ignored
              └─ Return WaveExecResult (kCompleted)
```

**BackendCaps:** `{timing: false, memory_trace: false, real_memory: true,
checkpoint: false}`

This is the fastest option by far — the real GPU runs at real speed. The
system simulator models everything around the wave execution (CP firmware
path, dispatch timing, fabric routing) but the shader work itself runs on
hardware.

**Constraints:**
- Requires real GPU hardware of ISA generation N (the transpiler's target)
- Memory coherence between simulator state and real GPU memory must be
  managed explicitly (sync simulated HBM ↔ real GPU memory before/after
  each kernel launch)
- No per-instruction observability — you can't single-step or inspect
  registers mid-kernel
- Memory trace is unavailable — the cache/memory hierarchy components in
  the system simulation won't see realistic traffic. For use cases that
  need memory modeling, fall back to backend 1 or 2
- Checkpoint/restore is limited because real GPU state is opaque

**When to use it:** Correctness validation of transpiled code at scale.
Multi-node simulation where wave-level detail doesn't matter but system-level
behavior (dispatch patterns, collective communication, fabric utilization)
does. Running real workloads through the simulated cluster topology to
validate software stack behavior.

### How the CU component selects a backend

The backend is configured per simulation, not per instruction. The topology
JSON config (or a command-line flag) selects the backend for each CU or
group of CUs:

```json
{
  "topology": {
    "root": {
      "name": "soc",
      "children": [{
        "name": "xcd[0:8]",
        "children": [{
          "name": "se[0:4]",
          "children": [{
            "name": "cu[0:8]",
            "config": [
              { "key": "wave_backend", "value": "fast-functional" }
            ]
          }]
        }]
      }]
    }
  }
}
```

You could even mix backends within a single simulation — cycle-accurate CUs
on one XCD for detailed analysis, fast-functional on the others for
background traffic generation. This is a standard technique in architecture
simulation (detailed core + simplified cores for traffic).

### What each team contributes

| Team | Contributes | Consumes from backplane |
|------|------------|------------------------|
| **Rocjitsu (Tony)** | simdojo, system model (CP, caches, memory controllers, fabric), cycle-accurate wave backend, RISC-V simulator, topology config, C API | ISA registry, code loading |
| **Mirage (Anush)** | Fast-functional wave backend (compiled-opcode interpreter, program cache), Python control plane, multi-arch ISA stubs | ISA registry, code loading, simdojo (as outer event shell or full integration) |
| **Hotswap** | Transpiler (ISA N+1 → N), real-GPU dispatch backend | ISA registry, code loading, wave backend interface |
| **Backplane (shared)** | ISA registry + codegen, code loading (ELF/offload bundles), wave backend interface, memory trace interface, hsakmt injection layer | AMD ISA XML specs |

### What changes for each project

**Rocjitsu:** Refactor `ComputeUnit::step()` to dispatch through
`WaveExecutionBackend`. The existing instruction execution becomes
backend 1. This is a mechanical refactor — extract the decode/execute path
into a class that implements the interface. The system model, simdojo
integration, topology config, and C API are unchanged.

**Mirage:** The `SingleGpuSimulator` and `Gfx950Interpreter` are refactored
into a library that implements `WaveExecutionBackend`. The compiled-opcode
pipeline (decode → compile → cache → switch-execute) stays exactly as-is.
What changes is the entry point: instead of being called from
`SingleGpuSimulator::ExecuteDispatch()`, it's called from rocjitsu's CU
component via the backend interface. The Python control plane can continue
to work by driving the rocjitsu simulation engine instead of the monolithic
simulator. Mirage's multi-arch ISA stubs (gfx1201, gfx1250) become backends
available to the system simulator for those architectures.

**Hotswap:** Implements `WaveExecutionBackend` with the transpiler as
`prepare()` and real GPU dispatch as `execute()`. This is new code, but
it's a clean, well-scoped implementation against a defined interface.

### The migration path

This doesn't require a big-bang rewrite. The steps are:

1. **Define the `WaveExecutionBackend` interface** in rocjitsu. Start with
   the cycle-accurate backend (extract from existing CU code). Ship it.
   Nothing else changes.

2. **Adapt mirage's interpreter as backend 2.** This can happen in parallel
   with step 1. The mirage team wraps their interpreter in the backend
   interface. Initially it runs standalone (as today). Once the interface
   exists in rocjitsu, plug it in.

3. **Add memory trace support to mirage's interpreter.** Instrument the
   load/store paths in the switch statement to emit `MemoryTraceEntry`
   records when a callback is registered. This is mechanical — every
   `case kGlobalLoadDword:` already has the address and size; just emit
   them.

4. **Build hotswap backend.** Depends on the transpiler existing. The
   backend itself is straightforward — marshal state, dispatch, unmarshal.
   The hard work is in the transpiler, which is a separate project.

5. **Share the ISA registry.** Converge the codegen pipelines so all
   backends decode the same way. This can happen in parallel with 1-4.

6. **Add rack-scale.** Once the single-GPU simulation works with pluggable
   backends, scaling to multi-GPU and multi-node is the distributed
   simdojo work described in Section 11.

## Appendix B: Simdojo Architecture (shared between both)

Simdojo implements conservative barrier-based PDES:

- **Component**: Active entity with named ports, lifecycle callbacks
- **CompositeComponent**: Container with child components
- **Port**: Directional connection point (IN/OUT) with protocol tags
- **Link**: Adjacency between ports with configurable latency
- **Simulation engine**: Priority-queue event processing with LBTS
  synchronization for multi-threaded execution
- **Execution modes**: Functional (synchronous, zero overhead) and
  Clocked (event-driven with propagation delay)
